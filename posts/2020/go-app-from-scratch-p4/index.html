In this post, we are going to set up a GitLab server on GCP. We need a git repository for our CI/CD pipeline. You might ask, there is a native GCP service called, Cloud Source Repository, why don't we just use that. Of course we can use that, but Source Repository lacks a lot of features like merge request, UI for code reviewing etc. In addition, we can learn how to set up our own GitLab Server and configure its domain and SSL, how cool is that. üòè

Create a custom VPC network

We need to create another custom VPC network for our GitLab server. This GitLab server stores our valuable code(lol), so if we need a isolate VPC network for further firewall & VPN configuration.

Create VPC network gitlab-network:

$ gcloud compute networks create gitlab-network \
    --bgp-routing-mode=global \
    --subnet-mode=custom
Created [https://www.googleapis.com/compute/v1/projects/[PROJECT_ID]/global/networks/gitlab-network].
NAME            SUBNET_MODE  BGP_ROUTING_MODE  IPV4_RANGE  GATEWAY_IPV4
gitlab-network  CUSTOM       GLOBAL

Create a subnet us-west1 in gitlab-network:

$ gcloud compute networks subnets create gitlab-subnet --network=gitlab-network --range=10.0.8.0/24 --region=us-west1
Created [https://www.googleapis.com/compute/v1/projects/[PROJECT_ID]/regions/us-west1/subnetworks/gitlab-subnet].
NAME           REGION    NETWORK         RANGE
gitlab-subnet  us-west1  gitlab-network  10.0.8.0/24

$ gcloud compute networks subnets list | grep gitlab-network
gitlab-subnet  us-west1                 gitlab-network  10.0.8.0/24

Create a Google Compute Engine(VM)

You can choose other machine types in GCP. Create a GCE instance on us-west1 region:

$ gcloud compute instances create gitlab-server \
    --zone us-west1-a \
    --machine-type n1-standard-2 \
    --network gitlab-network \
    --subnet gitlab-subnet \
    --image=ubuntu-1804-bionic-v20200610 \
    --image-project=ubuntu-os-cloud
Created [https://www.googleapis.com/compute/v1/projects/[PROJECT_ID]/zones/us-west1-a/instances/gitlab-server].
NAME           ZONE        MACHINE_TYPE   PREEMPTIBLE  INTERNAL_IP  EXTERNAL_IP   STATUS
gitlab-server  us-west1-a  n1-standard-2               10.0.8.2     [EXTERNAL_IP] RUNNING

Assign a Static external IP for GCE

By default, GCP assign a ephemeral IP address for our GCE instance. We need to promote our current IP address to Static if we want to set up a domain for our server.

$ gcloud compute addresses create gitlab-ip \
  --addresses [EXTERNAL_IP] \
  --region us-west1
Created [https://www.googleapis.com/compute/v1/projects/[PROJECT_ID]/regions/us-west1/addresses/gitlab-ip].

Create Firewall rule to allow SSH, HTTP(S)

Open port 22 for everyone, so that we can SSH to our GitLab server instance. You can configure --source-ranges to your office IP ranges or some other range.

$ gcloud compute firewall-rules create gitlab-ssh \
    --allow tcp:22 \
    --source-ranges 0.0.0.0/0 \
    --network gitlab-network

Created [https://www.googleapis.com/compute/v1/projects/[PROJECT_ID]/global/firewalls/gitlab-ssh].
NAME        NETWORK         DIRECTION  PRIORITY  ALLOW   DENY  DISABLED
gitlab-ssh  gitlab-network  INGRESS    1000      tcp:22        False

Open port 80,443, so we can access our GitLab server. Also, we usually will configure an IP ranges for access:

$ gcloud compute firewall-rules create gitlab-http \
    --allow tcp:80 \
    --source-ranges 0.0.0.0/0 \
    --network gitlab-network
Created [https://www.googleapis.com/compute/v1/projects/[PROJECT_ID]/global/firewalls/gitlab-http].

NAME         NETWORK         DIRECTION  PRIORITY  ALLOW   DENY  DISABLED
gitlab-http  gitlab-network  INGRESS    1000      tcp:80        False

$ gcloud compute firewall-rules create gitlab-https \
    --allow tcp:443 \
    --source-ranges 0.0.0.0/0 \
    --network gitlab-network
Created [https://www.googleapis.com/compute/v1/projects/[PROJECT_ID]/global/firewalls/gitlab-https].

NAME          NETWORK         DIRECTION  PRIORITY  ALLOW    DENY  DISABLED
gitlab-https  gitlab-network  INGRESS    1000      tcp:443        False

Set up a sub-domain for GitLab

We need to buy a domain for our GitLab server, or else we will access it using external IP of the GCE instance.

I bought my domain on GoDaddy, let's assume your domain is example.com. Set up a sub-domain gitlab.example.com to direct to [EXTERNAL_IP] that we reserved above.

After setting up our sub-domain, we can access to our GitLab page using the domain gitlab.example.com.

Not so fast
We haven't install our GitLab server yet. üòè

Install GitLab server

Follow the steps on this document to install GitLab. It might take some time to run through the installation, you should be fine. üòè

Make sure you use http://gitlab.example.com for external_url in /etc/gitlab/gitlab.rb, we'll set up SSL later.

After the installation is finished, we can access our GitLab with the external IP of the GCE. Open your browser with http://[EXTERNAL_IP], and we will see our GitLab login page.

Enable HTTPS for GitLab

Tip
  1. Make sure to create an A record for your gitlab domain.
  2. Make sure your 80 port is open through all of the firewall, including GCPs and OS level firewalls.
  3. Use https://letsdebug.net/ to troubleshoot Let's Encrypt HTTP-01 Challenge.

Since version 10.7, GitLab provided native Let's Encrypt intergration. We only need to add the following configuration in /etc/gitlab/gitlab.rb

external_url 'https://[GITLAB_DOMAIN]'
letsencrypt['enable'] = true
letsencrypt['auto_renew'] = true
letsencrypt['auto_renew_hour'] = 0
letsencrypt['auto_renew_minute'] = 30
letsencrypt['auto_renew_day_of_month'] = "*/4"
nginx['redirect_http_to_https'] = true
nginx['redirect_http_to_https_port'] = 80

After saving the file, execute the following command to re-configure GitLab:

$ gitlab-ctl reconfigure

Finising the re-configuration, we can browse to our GitLab page with https! And also we enabled http to https redirect, whenever there is a http access, it'll be redirect to https.
